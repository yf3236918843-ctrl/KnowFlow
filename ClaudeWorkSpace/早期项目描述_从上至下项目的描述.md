输入账号密码(可缓存,如果浏览器缓存了就不需要输入)并点击登录，或者注册。
登陆成功

if第一次使用无任何数据:新建一个学习项目,用自然语言描述这个项目的学习目标(例如微积分、线性代数...政治...哲学...等等)，在学习项目下新建题库，输入题库名,拍照上传(或者某些方式导入)

if有学习项目与题库,选择学习项目与题库,点击顺序做题->系统按上次做的题目的下一题出题,比如上次做了 第8面的第2题，那么这次就出第8面的第3题，用户也可以打开题目列表选择要做的题，题目列表里面能看到哪些做了哪些没做哪些生成了记忆哪些.....反正就是简单的浏览到信息,选择好一个题目后开始做题,正常的答疑,直到学生认为这道题已经差不多了,学生有两种选择 1.收录错题本 2.总结 if 收录错题本 -> 整个上下文会被发送给错题本草稿生成LLM，上下文进入"错题记录模式",在这个模式下,LLM可以给出具体的收录薄弱点,学生可以根据自己的想法用自然语言说,"薄弱点太少了有好多个薄弱点你都没收录" "这个薄弱点不需要" "根因不是这个"等,LLM会给出新的薄弱点生成草稿,直到学生在前端点击"确认收录"后，收录结束。 if 点击总结 -> 上下文会被发送给 "偏好总结"LLM 与 "习题情况记录"LLM 偏好总结会生成偏好，这些偏好会影响各个环节LLM的输出，例如答疑、偏好生成、错题收录、出题等等等等，例如学生在上下文中说了"不要罗里吧嗦的真的服了" "错题本怎么能这样生成呢？你要详细的列举每一个我的错点再来说啊,你怎么可能让我来列举，应该是你列举后，我来看哪些值得记录哪些不值得" "这题太难了,我压根就没做过多少次这种题,下次复习时你要考虑一下我的实际情况..."->总结时偏好总结LLM能看到并生成相应记录，用于下一次相似的场景。同时，这里也有一些技术性问题，两个偏好互相冲突怎么办?相似怎么办?等等....,解决办法是，把偏好像错题收录一样，让偏好生产时也产生一个"确认偏好"的那种框，用户可以说"不对，我不是这个意思，是出题的时候要尽量的看一下我的历史记录不要莫名其妙出很难的题"->生成新的偏好->用户再次确认->....->直到完成。但这里也还是有一些问题,一些"零散的偏好"如何记录?如果用户某次进行矫正,单独拎出来看不能被生成出一个偏好,但是多次相似的上下文进行了相似的矫正,其实就可以归纳出一种偏好，那这种零散的杂乱信号应该如何处理?。同时，生成偏好->用户审核->再生成... 这个循环中，也可以生成关于如何生成偏好的偏好。整个偏好系统目的就是"让系统在与用户的交流接触中慢慢磨合,最后实现你懂我,我懂你,越来越好用,你出的题刚好卡在我的边界区,你的回答我一看就懂而且学的很扎实,我一点击收录错题你就知道我想要收录哪些内容,极大的降低用户与系统的沟通成本"。 "习题情况记录"LLM 干的活就是，记录一下这题做的什么情况是对是错，生成整个上下文的summary(如何生成summary也是可以被偏好所影响的)等等有关于学习内容的事情。那么这就是"顺序做题"的大概功能。
复习功能:用户点击复习按钮->系统生成题目->用户做题->错题本收录或总结....

顺序注意:
错题本收录->总结 √
总结->错题本收录 ×
当用户点击错题本收录后，没有点击确认收录直接点击了总结->确认收录按钮灰掉不能点击无法收录
习题情况记录的内容无法通过自然语言修改。（你可以想想为什么我这样设计，用这个问题来跟你对齐一下想法的颗粒度）
你想想为什么会有这个顺序要注意？

当用户点击总结->自动跳转到下一题，就算summary 跟 偏好都还没有生成也是跳转，生成好了，如果有偏好生成，再用弹窗通知(或者小红点)

整个系统要实现的就是,用户只管"创建学习项目"与"导入题库"与"做题"，剩下的就交给系统，系统会根据他的情况来给他出题，让他可以沉浸在做题的心流里，不需要去考虑(还有哪些要复习?今天哪些不会做?.....哪些？。。。。哪些？....)
