# 头脑风暴完整总结（v2）

## 一、核心问题

项目缺少**用户端验收标准**——用户看到什么、点什么、发生什么。当前开发流程是"技术指导员写实施文档 → 开发员实现 → 测试员验证"，但没有一份文档定义"用户视角的正确行为是什么"。

---

## 二、Session 创建与恢复

### 创建参数

```
精确：项目.题组.题目
模糊：不传参 / 只传项目 / 只传项目+题组 → 后端按规则解析
```

### 恢复

只需 `sessionId`，session 文件自带类型信息，不需要额外传 `type`。

### 长期数据产出

| 数据 | 存储位置 | 说明 |
|------|---------|------|
| 题目状态（done/正确错误/总结内容） | 题库 | 影响后续选题 |
| 偏好（合并/更新/删除/新增/强化） | PreferenceEngine | 除新增外都影响现有偏好 |
| 错题薄弱点 | exercise_records | 影响后续诊断 |
| messages / extern / 卡片状态 | Session 文件 | 会话内数据 |

---

## 三、题显框

### 结构

```
[<] [题库 X/Y] 题目文本（单行截断） [📍跳转] [>]
```

### 行为

- 新增内题 → 右到左滚动动画（300ms，ease-in-out）
- 动画期间箭头不可点
- 边界：首题左箭头置灰，末题右箭头置灰，仅一题时都置灰
- 跳转按钮始终可点 → 对话区平滑滚动到该题首次出现位置
- 题目内容完全从 messages 中 `{"type":"question"}` JSON 推导，不走 extern

---

## 四、教师流程（聊天）

### 输入框

- Enter 发送，Shift+Enter 换行
- 支持粘贴图片
- 无字符限制
- 空消息/纯空格不可发送
- 发送后输入框禁用，发送按钮灰色
- AI 回复期间不可输入
- 网络断开 → 重试按钮
- 连续快速发送 → 阻止（除非手动停止生成）

### 流式渲染

- `/* */` → 聊天气泡，流式追加显示
- ` ```json ``` ` → 先显示"正在生成卡片..."气泡 → 解析成功后替换为对应卡片
- 解析失败 → 降级为普通文本气泡，显示原始内容

### 流中断处理

**不区分"正常"和"中断"。messages 有什么就渲染什么。**

- `/*你好*/` → 聊天气泡"你好"
- `/*你好，我是你的` → 聊天气泡"你好，我是你的"（没有闭合标记，当普通文本）
- ` ```json {"type":"xxx"` → 降级为普通文本气泡，显示原始内容

前端不做"这是不是中断了"的判断。messages 是唯一事实来源，渲染是纯函数。

### Think 气泡

- 收起：`思考中 (n 字)`，n 实时刷新，左侧 `▶`
- 展开：`▼` + think 内容区，等宽字体，实时追加
- 点击切换收起/展开
- 流结束后保留，可随时查看
- 恢复时默认收起
- **需改动**：`Session.stream()` 的 `finally` 块当前只存 `content`，需加 `thinking` 字段

## 五、偏好收集流程

### 触发

扩展菜单点"偏好收集" → 可在任意时刻触发，不限次数。

### 流程

```
点"偏好收集"
  → 前端发 switch_role(role: "preference_analyst")
  → 后端注入【系统指令】切换至角色:偏好分析师
  → 返回 role_switched
  → 前端：通知条"已进入偏好收录模式"、按钮变"继续做题"
  → 前端发 preference
  → SSE: think → output(粉色背景) → preference_actions 卡片
  → 用户勾选 → 乐观更新 DOM + extern_update 持久化
  → 用户点"确认" → confirm_preference
  → 通知条"已确认 N 项偏好"、卡片 disabled
  → 用户点"继续做题" → switch_role(role: "teacher") → 按钮恢复、粉色停止
```

### 规则

- 粉色背景仅限进入模式后新产生的 AI 气泡，历史气泡和用户气泡不变
- 每次触发生成新卡片，旧卡片状态不变
- 空卡片（0 项）→ 显示 0 项，不可确认
- 多次触发 → 多张卡片，各自独立确认

---

## 六、错题收录流程

同偏好收集，区别：

- 角色换成 `weakness_diagnostician`
- 卡片换成 `collect_draft`（多一个掌握度滑块 0-100）
- AI 气泡背景色：绿色（偏好是粉色）
- 颜色硬编码，不散落

---

## 七、总结流程

### 触发

扩展菜单点"总结本题" → 只能一次。

### 流程

```
点"总结本题"
  → 后端：查原题状态
    → 非"处女状态" → 返回错误，前端设总结按钮 done
    → 处女状态 → 标记原题 done → 创建新 session → 返回 {next_session_id}
  → 前端：直接跳转新 session（无动画）
  → 后端异步：
      注入总结员 prompt → 生成总结卡片
      → 注入偏好员 prompt → 生成偏好卡片
  → 旧 session 侧边栏小红点
  → 用户切回旧 session → 看到总结卡片 + 偏好卡片（可确认）
```

### 规则

- 题目 done 后：总结 JSON 永不重新处理（锁死）
- 仍可聊天、仍可偏好收录、仍可错题收录
- 总结按钮永久置灰
- done 前后的收录数据一视同仁，仅时间戳不同
- 无下一题 → 通知条"没有题了哦~"
- extern 存 `_question_ref`（创建时写入，永不更新），后端据此查原题状态

## 八、确认动作（通用规则）

### 防重复

连续两次点击确认 → 第一次未返回时弹通知条"正在处理"。所有类似功能都遵循此规则。

### 反馈

仅通知条（屏幕垂直居中、约 1/4 高度、几秒自动消失），对话区无额外气泡。

### 写入顺序

**永远先写入再报告完成，禁止先报告再执行写入。**

安卓风格的临时通知气泡。居中显示，自动消失。与对话区内的聊天气泡（bubble）是不同概念。

## 十、Pipeline Type 体系

前端统一发 `POST /api/pipeline/run`，body 为 `{pipeline: "exercise.<func>", params: {type?, ...}}`。

所有返回走 SSE，即使是一次性结果也包装成 SSE 事件。

### 完整 type 清单

| func | 触发 | 流式？ | 返回事件 |
|------|------|--------|---------|
| `start` | 新建做题 | ✅ | output（题目+初始tutor消息） |
| `chat` | 用户发消息 | ✅ | think → output → billing |
| `switch_role` | 点偏好/错题/继续做题 | ❌ | `{type: "role_switched", role: "..."}` |
| `collect` | 点"收录错题" | ✅ | think → output（诊断）→ 卡片 JSON |
| `preference` | 点"偏好收集" | ✅ | think → output（分析）→ 卡片 JSON |
| `confirm_collect` | 卡片点确认 | ❌ | `{type: "confirmed", count: N}` |
| `confirm_preference` | 卡片点确认 | ❌ | `{type: "confirmed", count: N}` |
| `summary` | 点"总结本题" | ✅ | 立即返回 `{next_session_id}`，后台异步生成 |
| `extern_update` | 卡片交互（勾选/滑块） | ❌ | `{type: "extern_updated"}` |
| `stop` | 手动停止生成 | ❌ | `{type: "stopped"}` |

### 关键设计

- `switch_role`：后端收到 `role: "xxx"` → 自己拼 `【系统指令】切换至角色:xxx` 注入 messages。前端不感知 prompt 格式。
- `extern_update`：用户勾选/拖滑块 → 前端乐观更新 DOM → 同时发 extern_update 持久化。
- `summary`：立即返回 next_session_id 让前端跳转，总结+偏好异步生成到旧 session。
- 所有操作走同一个 pipeline 函数，通过 type/object 区分处理。

---

## 十一、后端不维护"模式"状态

`_chat`、`_preference`、`_collect`、`_summary` 本质上都是往 messages 注入不同的 system prompt。LLM 自行决定输出什么。

- LLM 在"偏好员"模式下输出 collect_draft 也合法 → 前端照常渲染，用户照常确认
- 只有总结特殊：done 后总结 JSON 永不处理（锁死题目状态）

---

## 十二、系统指令过滤

角色切换指令（`【系统指令】切换至角色:xxx`、`【系统提示:xxx`）在 messages 里（LLM 需要看到），但用户不该看到。

**方案：前端集中前缀过滤。**

- 集中到一个函数、一个入口
- 前缀 `【系统指令】` 和 `【系统提示:` 稳定不变
- 之前脆弱是因为散落在三个地方各写各的

---

## 十三、extern 设计

### 定位

extern 是业务流程中产生的数据存储空间。**不存流式内容**。

**原因**：messages 的写入是原子的（`stream()` 的 `finally` 块流完成后一次性写入），extern 的写入是离散的（前端收到 chunk 就写）。流中断 → extern 有半截数据，messages 没有 → 恢复时两个数据源打架。

### 规则

- **messages 管"说了什么"**（后端原子写入）
- **extern 管"做了什么"**（前端离散写入：确认状态、done 标记、卡片交互）
- 两者不重叠

### 存储方式

按气泡 ID 存储。气泡 ID 由 message 唯一确定（重建时渲染引擎分配，同一 message 永远同一 ID）。

```
extern = {
  "_question_ref": "project_id.bank_id.group_id.question_id",  // 创建时写入，永不更新
  "bubble_abc123": { confirmed: true, items: [...] },
  "bubble_def456": { done: true },
}
```

`_question_ref` 是系统保留 key，创建 session 时写入，之后只读不写。

### 容错

extern 数据丢失 → 默认状态（如总结按钮认为"还没总结"，允许重新走流程）。可接受，属于意外情况。

## 十四、ViewManager 与扩展性

### 核心设计

```
栈（Stack）：一组 views 的数组，同一栈内 views 有层级关系（open/back）
后台池：多个栈并存，同一时间只有一个前台栈
```

### 操作

| 操作 | 行为 | 场景 |
|------|------|------|
| `show(factory, args)` | 匹配后台池 → 恢复；不匹配 → 新建栈 | 切换做题 session |
| `open(factory, args)` | 当前栈压入新 view，下层 suspend() | 做题中打开画图 |
| `back(result)` | 弹栈顶，下层 activate(ctx, result) | 画图完成，带回数据 |
| `vote(view, bool)` | 投票自毁，全栈同意 → 销毁 | 做题完成，清理旧 session |

### View 生命周期

```
create(args) → activate(ctx, result) → [suspend() → activate(ctx, result)] → deactivate()
```

`back(result)` 的 `result` 是跨 view 通信的唯一通道。

### 扩展工具（如画图）

```
做题中 → open(createDraw, {})
  → exercise view suspend()
  → 画图 view activate()
  → 用户画完确认
  → back({ image: "base64..." })
  → 画图 deactivate()，出栈
  → exercise view activate(ctx, { image: "base64..." })
  → exercise 拿到图片，发 chat 带图片
```

工具 view 不感知 session、不感知 pipeline，只负责产出数据，通过 `back(result)` 交还。

### 栈内通信（后续议题）

当前 `back(result)` 是单向一次性通信。后续需要栈内事件总线支持双向实时通信（如画图 view 实时显示 LLM 流式内容）。单独议题，不阻塞当前顺序做题验收标准。

---

## 十五、Session 恢复

完全从 Session 文件复原。`messages` 是唯一事实来源，渲染是纯函数 `f(messages, extern) → UI`。

- 题显框内容 → 从 messages 中所有 `{"type":"question"}` JSON 推导
- 卡片状态 → 从 extern 按气泡 ID 恢复
- 流未完成 → 前端重新建立 SSE 连接继续收（think 字数实时刷新）
- 恢复失败 → 通知条报错

---

## 十六、错误处理

全部走通知条，不走对话气泡（对话气泡内容必须能从 messages 推导）。

- 网络断开 → 重试按钮
- LLM 超时 → 通知条"AI 响应超时"
- API 报错 → 通知条 + 错误信息
- session.create 失败 → 通知条 + 停留当前页

---

## 十七、设计原则

1. **messages 是唯一事实来源**：渲染是纯函数，同样的 messages 产出同样的 UI
2. **extern 存业务状态，不存内容**：messages 管"说了什么"，extern 管"做了什么"
3. **模式是前端概念**：后端不维护"当前模式"，只负责注入角色切换 prompt
4. **工具 view 不感知业务**：产出数据 → back(result) → 调用方消费
5. **所有操作走 pipeline**：不新增 API 端点
6. **永远先写入再报告**：先持久化再通知用户
7. **流中断不区分**：messages 有什么渲染什么，前端不做"是不是中断了"的判断

## 文件 2：`exercise_workflow.py` — 完全重写

### 架构评估

**当前问题：**
- `run_stream` 的 if-elif 链线性增长，每加一个 func 加一个分支
- `_preference` 和 `_collect` 是 90% 相同的代码（get session → 注入 prompt → stream → 解析 → 存 extern → destroy）
- `_summary` 是同步的——在同一个 async generator 里等 LLM 生成完总结+偏好才返回，前端被阻塞
- `_start` 选题逻辑硬编码 `unfinished[0]`，不支持参数化选题
- 无 `switch_role`、`extern_update`、`stop`

**设计目标：**
- func 注册表替代 if-elif
- 提取公共的 session 生命周期模式
- summary 拆为同步响应 + 异步后台
- 选题支持精确/模糊参数

### 接口契约

#### `run_stream(ctx, sm, pm, pe, store, log)` — 不变

对外接口不变。内部改为注册表分发。

#### Func 注册表

```python
# 内部实现（不暴露给外部）
_FUNC_REGISTRY = {}

def _register_func(name):
    def decorator(fn):
        _FUNC_REGISTRY[name] = fn
        return fn
    return decorator
```

每个 func 签名统一：
```python
async def _func_name(ctx, user_id, sm, pm, pe, store, log) -> AsyncIterator[dict]:
```

#### 公共辅助：`_with_session(sid, user_id, sm, fn)`

```python
async def _with_session(sid, user_id, sm, fn):
    """get → fn(session) → destroy → yield"""
    session = sm.get(sid, user_id)
    if not session:
        yield {"type": "error", "message": "Session not found"}
        return
    try:
        async for chunk in fn(session):
            yield chunk
    finally:
        session.destroy()
```

所有需要 session 的 func 通过此辅助执行，消除重复的 get/destroy 代码。

#### 各 func 接口

**`start`**

```
输入：{bank_id, project_id?, group_id?, question_id?}
行为：
  1. 按参数选题（精确 > 模糊 > 默认取第一个未完成）
  2. 创建 session，extern 写入 _question_ref（格式："project_id.bank_id.group_id.question_id"）
  3. 注入 tutor prompt（append 到 messages，带前缀【系统指令】）
  4. destroy session
  5. yield question 事件
输出事件：
  {"type": "question", "question_id": N, "label": "...", "content": "...",
   "session_id": N, "question_list": [...]}
  {"type": "error", "message": "..."}  // 无题时
  {"type": "result", "message": "所有题目已完成"}
```

**`chat`**

```
输入：{session_id, message}
行为：
  1. get session
  2. 过滤用户消息中的【系统指令】前缀（防注入）
  3. session.stream([{"role": "user", "content": msg}])
  4. 透传 StreamChunk → think/output/billing 事件
  5. destroy session
输出事件：
  {"type": "think", "content": "..."}
  {"type": "output", "content": "..."}
  {"type": "billing", ...}
  {"type": "error", "message": "..."}
```

**`switch_role`**

```
输入：{session_id, role: "teacher" | "preference_analyst" | "weakness_diagnostician" | "summarizer"}
行为：
  1. get session
  2. 根据 role 拼【系统指令】切换至角色:xxx，append 到 messages
  3. destroy session
  4. yield role_switched
输出事件：
  {"type": "role_switched", "role": "preference_analyst"}
  {"type": "error", "message": "..."}
```

角色映射表（内部常量）：
```python
_ROLE_MAP = {
    "teacher": "教师",
    "preference_analyst": "偏好分析师",
    "weakness_diagnostician": "薄弱点诊断师",
    "summarizer": "总结员",
}
```

**`collect`**

```
输入：{session_id}
行为：
  1. get session
  2. 首次触发：注入完整格式 prompt（含 profile/extern）；非首次：注入简短切换指令
  3. session.stream([{"role": "user", "content": msg}])
  4. 透传 think/output
  5. 流结束后解析 JSON，提取 items，存入 extern.collect_rounds
  6. destroy session
输出事件：
  {"type": "think", "content": "..."}
  {"type": "output", "content": "..."}
  {"type": "billing", ...}
  {"type": "error", "message": "..."}
```

**`preference`**

```
输入：{session_id}
行为：同 collect，角色为 preference_analyst，存入 extern.pref_rounds
输出事件：同 collect
```

**`confirm_collect`**

```
输入：{session_id, round: N, items: [{id, active, mastery}]}
行为：
  1. get session
  2. 从 extern.collect_rounds[round] 取原始数据
  3. 如果已 confirmed → yield {"type": "error", "message": "已确认"}
  4. 合并前端 items 与原始数据（用 id 匹配，补 title/source/detail/types）
  5. 入库 exercise_records（仅 active=true 的项）
  6. 标记 confirmed=true
  7. append feedback 消息到 messages（【系统提示:薄弱点已存库】...）
  8. destroy session
  9. yield confirmed
输出事件：
  {"type": "confirmed", "count": N}
  {"type": "error", "message": "..."}
```

**`confirm_preference`**

```

输出事件：同 confirm_collect
```

**`summary`**

```
输入：{session_id}

  1. get session
  2. 从 extern._question_ref 查原题状态
  3. 如果原题已 done → yield error + destroy
  4. 标记原题 done（question_mark_done）
  5. 创建新 session（下一题）
  6. 立即 yield next_session_id
  7. destroy 旧 session

  {"type": "error", "message": "本题已总结"}         // 重复总结
  {"type": "error", "message": "..."}
```

注意：第 8-9 步是后台异步执行，不阻塞 yield。前端收到 `summary_started` 后立即跳转。后台完成后侧边栏小红点。

  4. yield extern_updated

**`stop`**

```
输入：{session_id}
行为：
  1. get session
  2. session.stop()  // 设置 stop_flag
  3. destroy session
  4. yield stopped

**流式管线行为：**

```
think chunk →
  如果当前无 think 气泡 → 创建 think 气泡（收起状态，显示 "思考中 (n 字)"）
  追加 think 内容，更新字数

done →
  封口当前气泡
  流结束

error →
  封口当前气泡
  显示错误通知条
```

**关键约束：**
- think 气泡在第一个 output chunk 到来之前可能已经存在
- 一个流可以产出：think → 文字 → 卡片 → 文字 → 卡片 → ... 任意顺序
- 卡片气泡替换"正在生成卡片..."气泡时无动画
- 流结束后 think 气泡保留，默认收起
