# 2026-06-13 顺序做题全量测试执行计划

## 1. 目标

这不是“随便点一遍”的冒烟测试，而是一份可重复执行的全量回归计划。

目标有 4 个：

1. 跑通 `顺序做题` 的主链路：`开始 -> 对话 -> 收录错题 -> 生成偏好 -> 总结本题 -> 跳转下一题 -> 恢复旧 session`。
2. 覆盖当前 UI 可直接触发的全部关键功能和关键边界。
3. 对每一步给出明确操作、明确预期 UI、明确落盘检查点。
4. 把“系统功能错误”和“模型/提示词表现偏差”分开记录，便于后续精准修复。

## 2. 当前系统边界

### 2.1 当前测试入口

- 页面入口：`http://127.0.0.1:5003/test/main.html`
- 默认登录账号：`test`
- 默认登录密码：`123456`
- 该账号对应用户：`user_id = 2`
- 页面会自动登录，不需要手工输入账号密码

### 2.2 本轮必须覆盖的 UI 功能

- 侧边栏 session 列表加载
- 点击 `顺序做题`
- 做题对话流式输出
- think 区展示

- `生成偏好`
- `总结本题`

- 旧 session 回看
- URL `session_id` 恢复
- 卡片确认后 disabled
- 数据写盘后再提示成功

### 2.3 需要特别对齐的 merge 边界

偏好 `merge` 现在要按下面这条理解执行测试：

- `merge` 不是单独的页面功能
- 它属于 `生成偏好` 这条链路中的一种合法输出
- 也就是说，当模型认为若干已有偏好需要合并时，应在偏好分析返回的 JSON 卡片里直接表达出来
- 前端应能把这种卡片正常渲染出来，用户应能正常确认

当前前后端已知相关形态：

- 偏好卡支持渲染 `source_entries`
- 后端 action 支持通过 `source_ids` 表达“本次更新/合并来自哪些旧偏好”
- 因此，`merge` 在当前版本里应作为 `生成偏好 -> action_set 卡片 -> 用户确认` 的一部分来测试

### 2.4 后端存在但当前 UI 没有单独入口的能力

以下能力不要误报成“前端功能缺失 bug”，因为当前版本没有正式 UI 入口：

- 独立的偏好管理界面
- 用户通过自然语言直接编辑历史偏好记录的专门界面

这些能力可以在报告里标记为：

- `当前版本无单独 UI 入口，未纳入独立页面测试`

### 5.1 文件检查目录

- session：`D:\FIles\documents\Project FIles\PT\Project\core\store\data\2\sessions\`
- 错题收录：`D:\FIles\documents\Project FIles\PT\Project\core\store\data\2\exercise_records\`
- 活跃偏好：`D:\FIles\documents\Project FIles\PT\Project\core\store\data\2\preferences_active\`

- 题目状态：`D:\FIles\documents\Project FIles\PT\Project\core\store\data\2\questions\`

### 5.2 严格区分两类问题

#### 系统功能 bug

- 页面打不开
- 侧边栏一直加载
- 发送后没有流式输出
- 后端已返回卡片 JSON，但前端没显示卡片
- 点击确认提示成功，但文件没写入
- 文件写入了，但卡片状态没锁定
- 总结后没有跳题
- 旧 session 回看失败

#### 模型/提示词偏差

- 你预期是 `update`，模型给了 `delete + insert`
- 你预期是 `empty`，模型还是给了一张 action_set 卡
- 你预期收录 2 条，模型只抽了 1 条

这类情况仍然要记录，但要标成：

- `模型输出偏差，系统流程本身仍可继续`

## 8. 关于偏好 merge 的专项说明

当前需要区分两种 merge：

### 8.1 本轮必须关注的 merge

- 在 `生成偏好` 时，模型直接返回 merge 风格的 JSON 卡片
- 这种 merge 属于当前 UI 测试范围内
- 一旦触发，就必须检查：
  - 卡片是否展示出来
  - 来源偏好是否能展开
  - 用户能否确认
  - 确认后文件是否正确落盘

### 8.2 当前没有单独 UI 入口的 merge

当前代码里弱信号自动合并阈值存在：

- 至少 3 条同类弱信号，或
- 最早弱信号超过 7 天

但当前版本没有一个单独的、稳定的页面入口去直接触发 `PreferenceEngine.try_merge()` 主动询问流，所以：

- 本计划不把“独立 merge 询问弹窗/页面”列为必过项
- 如果执行 AI 想补充观察，可以在报告中备注：
  - `后端具备 try_merge 能力，但当前版本未暴露独立稳定 UI 入口`

不要把这一点直接记成前端 bug。

## 10. 执行建议

1. 每次点完 `确认收录`，都立刻读磁盘，不要只看 UI。
2. 每次总结跳题后，先记下旧 `session_id` 和新 `session_id`。
3. 报 bug 时明确说清是：
   - `前端没显示`
   - `后端没写入`
   - `后端写了但前端状态没恢复`
   - `模型动作类型与预期不一致`
4. 如果某一步模型没有给出理想动作，不要立刻中断整条链路，先看系统是否仍能继续跑完。
5. 如果发现缓存疑似干扰前端结果，先强刷页面再复测一次，再决定是否报 bug。
